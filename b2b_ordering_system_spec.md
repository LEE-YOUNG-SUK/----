
# B2B 온라인 발주 시스템 구축 기술 사양서

## 1. 개요 (Overview)

본 문서는 기존 ERP 시스템 내에 각 지점(Clients)과 본사(동안기획) 간의 B2B 온라인 발주 시스템을 구축하기 위한 기술 사양을 정의합니다.

**핵심 목표:**
- **안정성:** 기존 시스템과 완벽히 분리되어 기능적 충돌이 없도록 설계합니다.
- **일관성:** 기존 코드베이스의 아키텍처, UI/UX, 코딩 컨벤션을 준수합니다.
- **확장성:** 단계별 구현을 통해 기능을 점진적으로 확장하고 유지보수가 용이한 구조를 지향합니다.

## 2. 핵심 아키텍처 및 규칙 (Core Architecture & Conventions)

새로운 기능은 다음의 기존 기술 스택과 설계 패턴을 따릅니다.

- **주요 기술 스택:** Next.js, Supabase, TypeScript, Tailwind CSS
- **데이터베이스 상호작용:**
    - 모든 데이터베이스 접근은 `lib/supabase/server.ts`의 서버 클라이언트를 통해 수행합니다.
    - 복잡한 데이터 처리(예: 배치 업데이트)는 필요시 Supabase의 RPC(PostgreSQL 함수)로 구현하여 일관성을 유지합니다.
- **UI/UX:**
    - 모든 UI 컴포넌트는 `components/ui/`에 있는 기존 원시 컴포넌트(`Button`, `Table`, `Dialog` 등)를 재사용하여 시각적 일관성을 확보합니다.
- **코드 구조:**
    - 신규 기능 관련 페이지는 `app/b2b-orders/` 디렉토리 하위에 구성합니다. (예: `app/b2b-orders/new`, `app/b2b-orders/history`)
    - 신규 기능 관련 컴포넌트는 `components/b2b-orders/` 디렉토리 하위에 구성합니다.
- **네이밍 컨벤션:**
    - 데이터베이스의 신규 테이블은 `b2b_` 접두사를 사용하여 기존 테이블과 명확히 구분합니다. (예: `b2b_orders`)

## 3. 데이터베이스 스키마 설계 (Database Schema Design)

Supabase 마이그레이션 기능을 사용하여 다음 테이블들을 생성합니다. RLS(Row Level Security)를 설정하여 데이터 접근 권한을 철저히 제어합니다.

- **참조:** `users` (기존), `clients` (기존, 사업자 정보가 있는 고객), `products` (기존)
- **신규 테이블:**

```sql
-- b2b_orders: 주문 기본 정보
CREATE TABLE public.b2b_orders (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    created_at timestamp with time zone DEFAULT now() NOT NULL,
    client_id bigint REFERENCES public.clients NOT NULL, -- 주문한 지점
    user_id uuid REFERENCES public.users NOT NULL, -- 주문한 사용자
    total_amount numeric DEFAULT 0 NOT NULL, -- 총 주문 금액
    status text DEFAULT 'pending' NOT NULL, -- 주문 상태: pending, confirmed, shipped, delivered, canceled
    memo text -- 주문 관련 메모
);
ALTER TABLE public.b2b_orders ENABLE ROW LEVEL SECURITY;
-- RLS 정책: 지점 사용자는 자신의 주문만, 본사 관리자는 모든 주문을 볼 수 있도록 설정.


-- b2b_order_items: 주문 상세 항목
CREATE TABLE public.b2b_order_items (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    order_id bigint REFERENCES public.b2b_orders ON DELETE CASCADE NOT NULL,
    product_id bigint REFERENCES public.products NOT NULL,
    quantity integer NOT NULL,
    unit_price numeric NOT NULL, -- 주문 시점의 단가
    total_price numeric GENERATED ALWAYS AS (quantity * unit_price) STORED
);
ALTER TABLE public.b2b_order_items ENABLE ROW LEVEL SECURITY;
-- RLS 정책: 연결된 b2b_orders의 정책을 따름.


-- b2b_transaction_statements: 거래명세서 정보
CREATE TABLE public.b2b_transaction_statements (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    created_at timestamp with time zone DEFAULT now() NOT NULL,
    order_id bigint REFERENCES public.b2b_orders NOT NULL,
    issued_at timestamp with time zone, -- 발송일
    statement_data jsonb, -- 명세서 내용 (공급자, 공급받는자, 품목, 수량, 금액 등)
    status text DEFAULT 'draft' NOT NULL -- 상태: draft, issued, canceled
);
ALTER TABLE public.b2b_transaction_statements ENABLE ROW LEVEL SECURITY;


-- b2b_tax_invoices: 세금계산서 정보
CREATE TABLE public.b2b_tax_invoices (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    created_at timestamp with time zone DEFAULT now() NOT NULL,
    order_id bigint REFERENCES public.b2b_orders NOT NULL,
    client_id bigint REFERENCES public.clients NOT NULL,
    request_at timestamp with time zone, -- 발행 요청일
    issued_at timestamp with time zone, -- 발행일
    provider_id text, -- 외부 연동 서비스에서의 고유 ID
    status text DEFAULT 'pending' NOT NULL -- 상태: pending, requested, issued, failed
);
ALTER TABLE public.b2b_tax_invoices ENABLE ROW LEVEL SECURITY;

```

## 4. 단계별 구현 계획 (Phased Implementation Plan)

### Phase 1: 기반 데이터 및 핵심 주문 기능

- **목표:** 지점 사용자가 상품 목록을 보고 주문을 생성하는 최소 기능 구현. 본사 관리자는 접수된 주문을 확인할 수 있어야 함.
- **DB 작업:**
    1. `b2b_orders` 테이블 생성.
    2. `b2b_order_items` 테이블 생성.
    3. 각 테이블에 RLS 정책 적용.
- **백엔드 (Server Actions in `app/b2b-orders/actions.ts`):**
    - `getProductsForOrder()`: 주문 가능한 상품 목록 조회 (기존 `products` 테이블 활용).
    - `createOrder(items: ...)`: 장바구니 정보를 받아 `b2b_orders` 및 `b2b_order_items`에 주문 데이터를 생성하는 RPC.
    - `getOrderHistory()`: 지점 사용자의 주문 내역 조회.
    - `getAllOrders()`: 본사 관리자용 모든 주문 내역 조회.
- **프론트엔드 (UI):**
    - **`app/b2b-orders/new/page.tsx`:** 상품 목록을 보여주고 장바구니에 담아 주문을 생성하는 페이지.
        - `components/b2b-orders/OrderForm.tsx`: 상품 선택, 수량 입력, 장바구니 기능 포함.
    - **`app/b2b-orders/history/page.tsx`:** 지점 사용자의 주문 내역 및 상태를 보여주는 테이블 페이지.
    - **`app/admin/b2b-orders/page.tsx`:** 본사 관리자용으로 접수된 모든 주문을 보여주는 테이블 페이지 (초기에는 조회만).

### Phase 2: 주문 처리 및 거래명세서

- **목표:** 본사 관리자가 주문 상태(`confirmed`, `shipped`)를 변경하고, `shipped` 상태일 때 거래명세서를 생성 및 발송하는 기능 구현.
- **DB 작업:**
    1. `b2b_transaction_statements` 테이블 생성 및 RLS 적용.
- **백엔드 (Server Actions 업데이트):**
    - `updateOrderStatus(orderId, status)`: 주문 상태를 변경하는 RPC.
    - `generateTransactionStatement(orderId)`: 주문 정보를 바탕으로 `b2b_transaction_statements`에 명세서 초안을 생성.
    - `issueTransactionStatement(statementId)`: 명세서를 '발행' 상태로 변경하고 PDF 생성 및 이메일 발송 로직 실행.
- **프론트엔드 (UI):**
    - **`app/admin/b2b-orders/page.tsx` 업데이트:**
        - 주문 목록 테이블에 상태 변경을 위한 드롭다운/버튼 추가.
        - '거래명세서 생성/보기' 버튼 추가.
    - **`components/b2b-orders/StatementDialog.tsx`:**
        - 거래명세서 내용을 보여주는 모달.
        - '이메일 발송', 'PDF 다운로드', '수정' 기능 버튼 포함.
    - **`app/b2b-orders/history/[orderId]/page.tsx`:** 주문 상세 페이지에서 생성된 거래명세서 조회 기능 추가.

### Phase 3: 세금계산서 및 보고서

- **목표:** 세금계산서 발행 요청 및 외부 연동 발행, 기본적인 매출 통계 조회 기능 구현.
- **DB 작업:**
    1. `b2b_tax_invoices` 테이블 생성 및 RLS 적용.
- **백엔드 (Server Actions 업데이트):**
    - `requestTaxInvoice(orderId)`: 세금계산서 발행을 요청. `b2b_tax_invoices`에 데이터 생성.
    - `issueTaxInvoice(taxInvoiceId)`: (외부 API 연동) 실제 전자세금계산서 발행 로직.
    - `getB2bSalesReport(period)`: 기간별 지점별 매출 통계를 집계하는 RPC.
- **프론트엔드 (UI):**
    - **`app/admin/b2b-tax-invoices/page.tsx`:** 세금계산서 발행 요청 목록 및 처리 상태 관리 페이지.
    - **`app/reports/b2b-sales/page.tsx`:** `getB2bSalesReport` 결과를 `Card`와 `Table` 컴포넌트를 이용해 보여주는 보고서 페이지.

## 5. 시스템 분리 및 안정성 확보 방안

- **독립된 경로 및 컴포넌트:** 모든 신규 UI는 `app/b2b-orders`, `components/b2b-orders` 등 독립된 디렉토리에서 관리하여 기존 코드와 물리적으로 분리합니다.
- **DB 스키마 분리:** 모든 신규 테이블에 `b2b_` 접두사를 붙여 기존 `purchases`, `sales` 테이블과의 명칭 및 논리적 충돌을 원천적으로 방지합니다.
- **공통 모듈의 신중한 사용:** UI 일관성과 개발 효율성을 위해 `components/ui`와 `lib/supabase/server.ts`는 재사용하되, 수정이 필요할 경우 하위 호환성을 반드시 고려합니다.
- **강화된 권한 제어:** 모든 신규 테이블에 대해 Supabase RLS 정책을 기본으로 적용합니다. 지점 사용자는 자신의 지점 데이터에만 접근할 수 있고, 본사 관리자는 모든 데이터에 접근할 수 있도록 역할을 명확히 구분합니다.
- **(선택) 기능 플래그(Feature Flag):** ` .env` 파일에 `NEXT_PUBLIC_B2B_ORDERING_ENABLED=true`와 같은 환경 변수를 추가하여, 관련 메뉴 노출 및 API 접근을 제어할 수 있습니다. 이를 통해 기능 초기 배포 시 안정성을 확보하고, 문제 발생 시 신속하게 기능을 비활성화할 수 있습니다.
